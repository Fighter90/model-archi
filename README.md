# Исходный процесс фитнес-клуба

## 1. Запись клиента на тренировку
- Клиент звонит по телефону или лично подходит к стойке администратора для записи на тренировку.
- Администратор вручную проверяет расписание тренировок, которое ведётся в бумажном журнале или на отдельных листах.
- Если нужное время свободно — администратор записывает клиента (в журнал, на лист бумаги, иногда в Excel-файл).
- В случае занятости времени — предлагаются альтернативные варианты, опять же на основе ручного поиска в бумажном расписании.
- Возможны ошибки: двойное бронирование, потеря/недостаточность информации о наличии свободных мест, неточности в расписании.

## 2. Формирование и ведение расписания
- Каждый тренер составляет своё расписание и передаёт его администратору (устно, по бумаге или через мессенджер).
- Администратор переносит информацию в общий бумажный журнал.
- При изменении расписания (перенос, отмена, добавление занятия) все изменения вносятся вручную.
- Часто случаются накладки — два тренера или группы могут быть записаны на один зал, информация о сменах теряется.

## 3. Приём и хранение информации о клиентах
- При первой записи клиента администратор заводит бумажную карточку или записывает данные в тетрадь.
- Хранятся минимальные данные: ФИО, номер телефона, тип абонемента, иногда отметки о медицинских ограничениях.
- Вся информация находится в бумажных архивах (коробки, папки, полки на стойке администратора).
- При повторных визитах клиента ищут по алфавиту в этих бумажных архивах.

## 4. Контроль посещаемости
- Посещение фиксируется вручную: администратор ставит отметку в журнале или на карточке клиента.
- Данные о посещаемости не систематизируются и не агрегируются.
- Нет простого способа посмотреть, сколько занятий осталось у клиента, сколько клиентов посетили клуб за период.

## 5. Работа с абонементами и оплатой
- Продажа абонемента фиксируется вручную — выдаётся бумажная квитанция или делается отметка в журнале.
- Нет единой базы по действующим абонементам, их срокам, истечению и количеству оставшихся посещений.
- В случае утери бумажной информации восстановить данные невозможно или очень сложно.

## 6. Уведомления и коммуникация с клиентами
- Напоминания о тренировках не отправляются.
- Для связи с клиентом администратор звонит вручную по телефону, если есть необходимость отменить или перенести занятие.
- Часто происходят случаи, когда клиент забывает о тренировке или приходит на отменённое занятие.

## 7. Аналитика и управленческая отчётность
- Все показатели собираются вручную (если собираются вообще).
- Для руководства трудно получить точные данные о загрузке клуба, посещаемости, продажах абонементов, причинах оттока клиентов.
- Большая часть управленческих решений принимается на основе субъективных наблюдений, а не фактических данных.

---

## Участники процесса (Business Roles/Actors)
- **Клиент** — физическое лицо, пользующееся услугами клуба.
- **Администратор** — сотрудник, осуществляющий запись, ведение расписания, коммуникации с клиентами.
- **Тренер** — сотрудник, проводящий занятия, формирующий своё расписание.
- **Руководство клуба** — использует собранные данные для принятия решений (опционально, если используется ручная отчётность).

---

## Объекты данных (Business/Data Objects)
- **Бумажное расписание занятий** — журнал или листы с текущим расписанием групп/тренеров/зал.
- **Карточка клиента** — бумажная форма, содержащая ФИО, телефон, абонемент, количество посещений, иногда — особые отметки.
- **Журнал посещаемости** — бумажный документ для отметок о фактическом посещении.
- **Журнал продаж абонементов** — бумажная ведомость по продажам, оплатам.
- **Заметки/сообщения тренеров** — расписания и пожелания тренеров (на бумаге или через мессенджер).
- **Телефон клиента** — используется для звонков и ручной коммуникации.
- **Журнал инцидентов/отказов** (не всегда ведётся) — фиксирует случаи конфликтов по расписанию или другие инциденты.

---

## Информационные потоки (Business/Data Flows)

### Запись на тренировку
- Клиент → (по телефону/лично) → Администратор
    - Сообщает свои пожелания по времени, типу занятия.
- Администратор ↔ Бумажное расписание
    - Проверяет доступность времени, вносит запись.
- Администратор → Карточка клиента
    - Фиксирует новую запись или создает карточку, если клиент новый.

### Ведение расписания
- Тренер → Администратор
    - Передаёт своё расписание и пожелания.
- Администратор → Бумажное расписание
    - Вносит изменения, обновляет расписание.

### Контроль посещаемости
- Клиент → Администратор
    - Сообщает о прибытии.
- Администратор → Журнал посещаемости/Карточка клиента
    - Ставит отметку о посещении, списывает занятие с абонемента.

### Продажа абонементов
- Клиент → Администратор
    - Оформляет покупку абонемента.
- Администратор → Журнал продаж абонементов / Карточка клиента
    - Фиксирует факт оплаты, количество оставшихся занятий.

### Коммуникация с клиентом
- Администратор → (Звонок) → Клиент
    - Информирует об изменениях, отменах, напоминаниях (по необходимости).

### Отчётность
- Администратор → Руководство
- Передаёт (устно/на бумаге) обобщённые данные по посещаемости, продажам, загруженности.

---

## Отдельные события и проблемы (Issues)
- Потеря или порча бумажных данных.
- Неактуальность информации из-за задержки ручного обновления.
- Ошибки двойных записей или пропущенных клиентов.
- Проблемы коммуникации между тренерами и администраторами.

# Описание текущей модели (AS-IS) в ArchiMate (Клиенты) по моделе AS-IS Fitness Club — Integrated View
```
На основе ArchiMate модели фитнес-клуба, вот как работают основные процессы:

## Процесс записи клиентов (Booking)

**Инициация:** Клиент приходит в клуб (`event-client-comes`)

**Основной поток:**
1. **Проверка слота** (`proc-slot-check`) - проверяется доступность места на занятие через бумажное расписание (`obj-paper-schedule`)
2. **Резервирование** (`proc-reserve`) - если место свободно, оно резервируется с использованием телефонных данных клиента (`obj-client-phone`)
3. **Альтернативное предложение** (`proc-offer-alt`) - если место занято, предлагаются альтернативные варианты

**Участники:** Клиент через интерфейс стойки (`iface-desk`), администратор стойки (`role-frontdesk`)

## Процесс управления карточками клиентов (Card Management)

**Основные операции:**
1. **Регистрация** (`proc-register`) - создание новой карточки клиента (`obj-client-card`)
2. **Поиск карточки** (`proc-find-card`) - поиск существующей карточки
3. **Чек-ин** (`proc-check-in`) - отметка посещения с обновлением баланса посещений (`obj-visit-balance`) и журнала посещений (`obj-attendance-log`)

**Участники:** Администратор стойки, тренер

## Процесс планирования расписания (Schedule Planning)

**Компоненты:**
1. **Расписание тренера** (`proc-trainer-schedule`) - тренер создает свое расписание с заметками (`obj-trainer-notes`)
2. **Основное расписание** (`proc-main-schedule`) - администрация формирует общее расписание
3. **Синхронизация** (`interact-schedule-sync`) - согласование между тренерами и администрацией

**События:** Изменения расписания (`event-schedule-change`) могут привести к отмене занятий (`event-class-cancel`)

## Процесс отчетности (Reporting)

**Инициация:** Запрос отчета (`event-report-request`) от руководства

**Выполнение:**
1. **Генерация отчета** (`proc-report`) - аналитика данных посещений, карточек клиентов, расписания
2. **Создание документа** (`event-report-gen`) - формирование итогового отчета

**Источники данных:** Журналы посещений, карточки клиентов, баланс посещений, расписание

## Интеграционные связи

- Все процессы связаны через роль администратора стойки (`role-frontdesk`)
- Данные синхронизируются между различными объектами (карточки, расписания, журналы)
- Сервисы (`svc-booking`, `svc-card-mgmt`, `svc-attendance`) обеспечивают автоматизацию процессов
- Клиенты взаимодействуют через единый интерфейс стойки (`iface-desk`)

Система работает преимущественно на бумажных носителях с минимальной автоматизацией через сервисы.
```

# Детализация клиентских процессов (AS-IS)

## Алгоритм процесса "Бронирование места клиентом" (proc-client-booking)

```
1. НАЧАЛО: Клиент приходит (event-client-comes)
2. Проверка доступности слота (proc-slot-check):
   - Получить данные из бумажного расписания (obj-paper-schedule)
   - ЕСЛИ слот свободен → перейти к шагу 3
   - ЕСЛИ слот занят → предложить альтернативы (proc-offer-alt)
3. Резервирование места (proc-reserve):
   - Записать в расписание (obj-paper-schedule) 
   - Получить контактные данные (obj-client-phone)
   - Подтвердить бронирование (event-booking-confirmed)
4. КОНЕЦ: Уведомить клиента (proc-manual-notify)
```

## Алгоритм процесса "Регистрация клиента" (proc-register)

```
1. НАЧАЛО: Администратор запускает регистрацию
2. Создать карту клиента:
   - Записать данные в карточку клиента (obj-client-card)
   - Получить номер телефона (obj-client-phone)
3. Активировать услугу управления картами (svc-card-mgmt)
4. Перейти к продаже абонемента (proc-sell-subscription)
5. КОНЕЦ
```

## Алгоритм процесса "Поиск карты клиента" (proc-find-card)

```
1. НАЧАЛО: Администратор ищет карту
2. Найти карточку клиента (obj-client-card)
3. Активировать услугу управления картами (svc-card-mgmt)
4. Перейти к регистрации посещения (proc-check-in)
5. КОНЕЦ
```

## Алгоритм процесса "Регистрация посещения" (proc-check-in)

```
1. НАЧАЛО: От поиска карты или обновления статуса
2. Проверить данные:
   - Статус клиента (obj-client-status)
   - Баланс посещений (obj-visit-balance)
   - Карточка клиента (obj-client-card)
3. Записать посещение (obj-attendance-log)
4. Активировать услугу посещаемости (svc-attendance)
5. Сгенерировать запрос отчёта (event-report-request)
6. КОНЕЦ
```

## Алгоритм процесса "Составление расписания тренера" (proc-trainer-schedule)

```
1. НАЧАЛО: Тренер или администратор создаёт расписание
2. Записать в заметки тренера (obj-trainer-notes)
3. Активировать услугу расписания (svc-schedule)
4. Передать в основное расписание (proc-main-schedule)
5. КОНЕЦ
```

## Алгоритм процесса "Основное расписание" (proc-main-schedule)

```
1. НАЧАЛО: От расписания тренера
2. Обновить бумажное расписание (obj-paper-schedule)
3. Активировать услугу расписания (svc-schedule)
4. Сгенерировать изменение расписания (event-schedule-change)
5. КОНЕЦ
```

## Алгоритм процесса "Генерация отчётов" (proc-report)

```
1. НАЧАЛО: По запросу отчёта (event-report-request)
2. Собрать данные из:
   - Карточки клиентов (obj-client-card)
   - Журнала посещений (obj-attendance-log)
   - Баланса посещений (obj-visit-balance)
   - Бумажного расписания (obj-paper-schedule)
   - Кассы (obj-cashbox)
   - Чеков (obj-receipt)
3. Сгенерировать отчёт (event-report-gen)
4. КОНЕЦ
```

## Алгоритм процесса "Продажа абонемента" (proc-sell-subscription)

```
1. НАЧАЛО: После регистрации клиента
2. Получить номер телефона клиента (obj-client-phone)
3. Создать абонемент с истечением срока (event-abn-expiry)
4. Перейти к выдаче чека (proc-receipt)
5. КОНЕЦ: Уведомить об истечении (proc-manual-notify)
```

## Алгоритм процесса "Выдача чека" (proc-receipt)

```
1. НАЧАЛО: После продажи абонемента
2. Создать чек (obj-receipt)
3. Зафиксировать получение платежа (event-payment-received)
4. Перейти к сверке (proc-reconcile)
5. КОНЕЦ
```
