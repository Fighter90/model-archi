Проект должен включать:
- Модель мотивации для целевого состояния
- Архитектуру бизнеса (текущее и целевое состояние)
- Архитектуру информационных систем (текущее и целевое состояние)
- Архитектуру технической инфраструктуры (текущее и целевое состояние)
- Модель перехода из текущего состояния в целевое

---

## Качественные критерии оценки

- Все модели выполнены в одном Archi-файле
- Диаграммы используют **все элементы и связи** ArchiMate
- Диаграммы корректно применяют язык ArchiMate
- Диаграммы логичны по сути и визуально читаемы
- Используется цветовое кодирование для выделения
- Между диаграммами есть прослеживаемые связи

- **Модель мотивации:**
    - Одна диаграмма, не менее **70 элементов**

- **Архитектура бизнеса:**
    - Шесть диаграмм (по три для текущего и целевого состояния)
    - В каждой диаграмме не менее **50 элементов**

- **Архитектура информационных систем:**
    - Шесть диаграмм (по три для текущего и целевого состояния)
    - В каждой диаграмме не менее **40 элементов**

- **Архитектура технической инфраструктуры:**
    - Шесть диаграмм (по три для текущего и целевого состояния)
    - В каждой диаграмме не менее **40 элементов**

- **Модель перехода:**
    - Три диаграммы (по одной на каждую область)
    - В каждой диаграмме не менее **30 элементов**

# Исходный процесс фитнес-клуба

## 1. Запись клиента на тренировку
- Клиент звонит по телефону или лично подходит к стойке администратора для записи на тренировку.
- Администратор вручную проверяет расписание тренировок, которое ведётся в бумажном журнале или на отдельных листах.
- Если нужное время свободно — администратор записывает клиента (в журнал, на лист бумаги, иногда в Excel-файл).
- В случае занятости времени — предлагаются альтернативные варианты, опять же на основе ручного поиска в бумажном расписании.
- Возможны ошибки: двойное бронирование, потеря/недостаточность информации о наличии свободных мест, неточности в расписании.

## 2. Формирование и ведение расписания
- Каждый тренер составляет своё расписание и передаёт его администратору (устно, по бумаге или через мессенджер).
- Администратор переносит информацию в общий бумажный журнал.
- При изменении расписания (перенос, отмена, добавление занятия) все изменения вносятся вручную.
- Часто случаются накладки — два тренера или группы могут быть записаны на один зал, информация о сменах теряется.

## 3. Приём и хранение информации о клиентах
- При первой записи клиента администратор заводит бумажную карточку или записывает данные в тетрадь.
- Хранятся минимальные данные: ФИО, номер телефона, тип абонемента, иногда отметки о медицинских ограничениях.
- Вся информация находится в бумажных архивах (коробки, папки, полки на стойке администратора).
- При повторных визитах клиента ищут по алфавиту в этих бумажных архивах.

## 4. Контроль посещаемости
- Посещение фиксируется вручную: администратор ставит отметку в журнале или на карточке клиента.
- Данные о посещаемости не систематизируются и не агрегируются.
- Нет простого способа посмотреть, сколько занятий осталось у клиента, сколько клиентов посетили клуб за период.

## 5. Работа с абонементами и оплатой
- Продажа абонемента фиксируется вручную — выдаётся бумажная квитанция или делается отметка в журнале.
- Нет единой базы по действующим абонементам, их срокам, истечению и количеству оставшихся посещений.
- В случае утери бумажной информации восстановить данные невозможно или очень сложно.

## 6. Уведомления и коммуникация с клиентами
- Напоминания о тренировках не отправляются.
- Для связи с клиентом администратор звонит вручную по телефону, если есть необходимость отменить или перенести занятие.
- Часто происходят случаи, когда клиент забывает о тренировке или приходит на отменённое занятие.

## 7. Аналитика и управленческая отчётность
- Все показатели собираются вручную (если собираются вообще).
- Для руководства трудно получить точные данные о загрузке клуба, посещаемости, продажах абонементов, причинах оттока клиентов.
- Большая часть управленческих решений принимается на основе субъективных наблюдений, а не фактических данных.

---

## Участники процесса (Business Roles/Actors)
- **Клиент** — физическое лицо, пользующееся услугами клуба.
- **Администратор** — сотрудник, осуществляющий запись, ведение расписания, коммуникации с клиентами.
- **Тренер** — сотрудник, проводящий занятия, формирующий своё расписание.
- **Руководство клуба** — использует собранные данные для принятия решений (опционально, если используется ручная отчётность).

---

## Объекты данных (Business/Data Objects)
- **Бумажное расписание занятий** — журнал или листы с текущим расписанием групп/тренеров/зал.
- **Карточка клиента** — бумажная форма, содержащая ФИО, телефон, абонемент, количество посещений, иногда — особые отметки.
- **Журнал посещаемости** — бумажный документ для отметок о фактическом посещении.
- **Журнал продаж абонементов** — бумажная ведомость по продажам, оплатам.
- **Заметки/сообщения тренеров** — расписания и пожелания тренеров (на бумаге или через мессенджер).
- **Телефон клиента** — используется для звонков и ручной коммуникации.
- **Журнал инцидентов/отказов** (не всегда ведётся) — фиксирует случаи конфликтов по расписанию или другие инциденты.

---

## Информационные потоки (Business/Data Flows)

### Запись на тренировку
- Клиент → (по телефону/лично) → Администратор
    - Сообщает свои пожелания по времени, типу занятия.
- Администратор ↔ Бумажное расписание
    - Проверяет доступность времени, вносит запись.
- Администратор → Карточка клиента
    - Фиксирует новую запись или создает карточку, если клиент новый.

### Ведение расписания
- Тренер → Администратор
    - Передаёт своё расписание и пожелания.
- Администратор → Бумажное расписание
    - Вносит изменения, обновляет расписание.

### Контроль посещаемости
- Клиент → Администратор
    - Сообщает о прибытии.
- Администратор → Журнал посещаемости/Карточка клиента
    - Ставит отметку о посещении, списывает занятие с абонемента.

### Продажа абонементов
- Клиент → Администратор
    - Оформляет покупку абонемента.
- Администратор → Журнал продаж абонементов / Карточка клиента
    - Фиксирует факт оплаты, количество оставшихся занятий.

### Коммуникация с клиентом
- Администратор → (Звонок) → Клиент
    - Информирует об изменениях, отменах, напоминаниях (по необходимости).

### Отчётность
- Администратор → Руководство
- Передаёт (устно/на бумаге) обобщённые данные по посещаемости, продажам, загруженности.

---

## Отдельные события и проблемы (Issues)
- Потеря или порча бумажных данных.
- Неактуальность информации из-за задержки ручного обновления.
- Ошибки двойных записей или пропущенных клиентов.
- Проблемы коммуникации между тренерами и администраторами.

# Описание текущей модели (AS-IS) в ArchiMate (Клиенты) по моделе AS-IS Fitness Club — Integrated View

# Анализ архитектурной модели фитнес-клуба (AS-IS)

## Обзор модели

Модель описывает текущее состояние (AS-IS) системы управления фитнес-клубом, разделенную на три основные области:
- **Запись (Booking)** - процессы бронирования и записи клиентов
- **Хранение данных (Storage)** - управление данными клиентов и расписанием
- **Отчётность (Reporting)** - аналитика и формирование отчетов

## Основные действующие лица

### Внешние акторы
- `actor-client` - Клиент фитнес-клуба
- `actor-admin` - Администратор системы
- `actor-trainer` - Тренер
- `actor-management` - Руководство клуба

### Роли
- `role-client` - Роль клиента в системе
- `role-frontdesk` - Роль администратора стойки регистрации
- `role-trainer` - Роль тренера
- `role-management` - Роль руководства

## Область: Запись (Booking)

### Сервисы
- `svc-booking` - Сервис бронирования занятий
- `svc-schedule` - Сервис управления расписанием

### Ключевые процессы
- `proc-client-booking` - Основной процесс записи клиента
    - `proc-slot-check` - Проверка доступности слота
    - `proc-reserve` - Резервирование места

### События и связи
event-client-comes → proc-slot-check → proc-reserve

### Объекты данных
- `obj-paper-schedule` - Бумажное расписание
- `obj-client-phone` - Телефон клиента
- `product-pass` - Абонемент клиента

### Интерфейсы
- `iface-desk` - Интерфейс стойки регистрации

## Область: Хранение данных (Storage)

### Сервисы
- `svc-card-mgmt` - Управление карточками клиентов
- `svc-attendance` - Учет посещаемости

### Функции
- `func-records` - Функция ведения записей
- `func-planning` - Функция планирования

### Процессы управления данными
- `proc-register` - Регистрация нового клиента
- `proc-find-card` - Поиск карточки клиента
- `proc-check-in` - Регистрация прихода клиента
- `proc-trainer-schedule` - Планирование тренерского расписания
- `proc-main-schedule` - Основное планирование расписания

### Объекты данных
- `obj-client-card` - Карточка клиента
- `obj-attendance-log` - Журнал посещений
- `obj-visit-balance` - Баланс посещений
- `obj-trainer-notes` - Заметки тренера
- `obj-client-status` - Статус клиента
- `obj-client-phone` - Телефон клиента

### Цепочки процессов

proc-register → svc-card-mgmt proc-find-card → proc-check-in → svc-attendance proc-trainer-schedule → proc-main-schedule

## Область: Отчётность (Reporting)

### Сервисы
- `svc-reporting` - Сервис формирования отчетов

### Функции
- `func-analytics` - Аналитические функции

### Процессы
- `proc-report` - Формирование отчетов

### События
event-report-request → proc-report → event-report-gen

## Ключевые связи между областями

### Триггерные связи (TriggeringRelationship)
- Клиент приходит → проверка слота → резервирование
- Изменение расписания → уведомления
- Запрос отчета → генерация отчета

### Связи назначения (AssignmentRelationship)
- Тренеры назначаются на планирование расписания
- Администратор стойки назначается на проверку баланса

### Связи доступа (AccessRelationship)
- Процессы обновляют и читают данные клиентов
- Отчеты используют данные посещений и балансов

### Связи реализации (RealizationRelationship)
- Функции реализуют сервисы
- Процессы реализуют функциональность сервисов

### Связи обслуживания (ServingRelationship)
- Сервисы обслуживают роли пользователей
- Локации предоставляют интерфейсы

## Агрегационные группы

### Группа: Запись (`grp-booking`)
Содержит элементы, связанные с процессом бронирования и записи клиентов.

### Группа: Хранение (`grp-storage`)
Объединяет элементы управления данными, включая процессы обновления статуса, планирования и разрешения конфликтов.

### Группа: Отчётность (`grp-reporting`)
Включает элементы продаж, формирования чеков и уведомлений.

## Коллаборации
- `collab-admin-trainers` - Взаимодействие администраторов и тренеров
- `interact-schedule-sync` - Синхронизация расписаний

## Контракты и продукты
- `contract-sub` - Контракт на абонемент
- `product-pass` - Продукт "Абонемент"

## Основные потоки данных

1. **Поток записи клиента:**
   Клиент → Проверка слота → Резервирование → Подтверждение

2. **Поток управления данными:**
   Регистрация → Карточка клиента → Проверка баланса → Регистрация прихода

3. **Поток отчетности:**
   Запрос отчета → Обработка данных → Генерация отчета
